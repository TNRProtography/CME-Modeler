Spot The Aurora

Technical documentation — pages, backend workers, data sources, and scoring algorithms.

Stack: React 18 + Vite PWA on Cloudflare Pages. Seven Cloudflare Workers handle scoring, data processing, push notifications, and sightings. Three.js (CME modeller), Leaflet (sightings map), Chart.js (charts), Tailwind CSS. Font: Inter variable.

🌌 CME Modeller Page

3D solar system simulation with real NASA DONKI CMEs modelled using the Drag-Based Model. Built with Three.js r128.

🔭 3D Simulation

Top-down and side views of the inner solar system. CMEs are expanding arc meshes. Coronal holes overlaid from NOAA SUVI 195Å analysis.

Propagation Model (DBM)

  • Inner boundary: 21.5 solar radii (~0.1 AU)
  • Background slow solar wind: 380 km/s default
  • Drag parameter γ estimated from CME speed and half-angle
  • Solar rotation at synodic velocity 2.618×10⁻⁶ rad/s
  • CME–CME interactions flagged: preconditioning, compression, cannibalism
  • Coronal hole HSS and SIR streams modelled separately

Coronal Hole Detection

Client-side SUVI 195Å brightness thresholding detects coronal holes (dark = open field lines = fast solar wind). Snapshots stored in the CH History Worker for evolution tracking up to 72 hours.

⚙️ Worker Details

🌐 SpotTheAurora Forecast API

https://spottheaurora.thenamesrock.workers.dev/

Main scoring engine. Cron trigger every 5 minutes. Results stored in KV and served from Cloudflare edge cache (4-minute TTL). Fetch requests are near-instant — no computation on each request.

Data sources

SourceWhat it fetches
NOAA OVATIONSouthern hemispheric auroral power in GW — primary scoring input
NOAA RTSW mag 1mReal-time Bt and Bz — used for Bz score boost calculation
OpenWeatherMap One Call v3Sunrise, sunset, moonrise, moonset, moon phase — KV cached 6 hours

Scoring algorithm (v25)

  • Latitude-adjusted threshold: hpFor100 = 200 + (userLat − −42.45) × 35, clamped 120–450 GW. Power score = (HP / hpFor100) × 100
  • Bz boost: Linear 0–40 point bonus as Bz goes from 0 to −20 nT southward. Zero for northward Bz.
  • Base score: clamp(powerScore + bzBoost, 0, 100)
  • Moon reduction: 0.75 GW per 1% illumination when moon is up (with 10-minute rise/set ramps). Score deduction = moonReductionGW × 0.5
  • Time-of-day ramp: Ramps in 30 min before sunset, ramps out 40 min before sunrise. Zero during daytime.
  • South bright-moon clamp: Locations south of Greymouth + moon >60% illumination + moon up → score forced to 0.
  • Rolling average: 30-minute rolling average of raw and final scores stored to smooth short-term spikes.

KV storage keys

  • v1_api_payload — full API response object
  • raw_history — 24h of per-5-min raw score records
  • averaged_history — 24h of 30-min rolling averaged scores
  • daily_sun_moon_history — last 4 days of sun/moon data
  • latest_scores — current scores snapshot for push worker
  • cached_owm_data_v1 — OWM response, refreshed every 6 hours

⚡ Substorm Risk Worker

https://aurora-index-sta.thenamesrock.workers.dev/api/substorm

Real-time substorm scoring engine based on space physics. Fetches live data on each request, caches response 60 seconds. Available at ?resolution=1m (full 24h history) or 5m, 10m, 15m (downsampled).

Data sources

SourceWhat it fetches
GeoNet Tilde — EY2MSigned dH (nT/min) at Eyrewell magnetometer, Canterbury. 1-minute resolution, last 24h.
NOAA solar-wind mag-1-dayBz (GSM) and Bt from DSCOVR at L1, 1-minute resolution
NOAA solar-wind plasma-1-daySolar wind density, speed, temperature from DSCOVR at L1

Processing pipeline

  • Signed dH integration: Cumulative sum of signed dH reconstructs relative H-component trace (de-meaned over 24h) for bay-depth detection — no separate H endpoint needed
  • Sq baseline subtraction: 60-minute rolling median of |dH| computed; half subtracted from each point to remove the regular diurnal Sq variation that peaks near local magnetic noon
  • Timeline merge: All three streams aligned to GeoNet 1-minute timeline by forward-filling NOAA data to each GeoNet minute
  • Score per minute: Computed across the full 24h merged series and downsampled to requested resolution

CME/sheath flag: Richardson-Paularena temperature ratio (observed/expected). If ratio <0.5 (cold ejecta) or >4.0 (sheath turbulence), a 10% solar score boost is applied — these intervals commonly produce the strongest substorms.

🛰️ IMAP Solar Wind Proxy

https://imap-solar-data-test.thenamesrock.workers.dev/rtsw/merged-24h

Merges two solar wind data sources into a unified 24-hour per-minute grid. Each field independently prefers DSCOVR RTSW, falling back to IMAP I-ALiRT on gaps or sentinel values.

Sources (priority order)

SourceFieldsNotes
DSCOVR RTSW Operational (primary)speed, density, temp, Bx, By, Bz, Bt, angleNOAA plasma-1-day + mag-1-day
IMAP I-ALiRT (fallback)speed, density, temp (SWAPI), Bx, By, Bz, Bt, angle (MAG)Adaptive chunk fetching. Optional API key via env var.

Key features

  • Per-field fallback: Bz, speed, etc. each independently pick their best available source minute-by-minute
  • Sentinel filtering: Both sources filter −9999.9, −999.9, −1×10⁵, ≤−1×10³⁰ to null before fallback — sentinel values never block the other source
  • IMAP lookback: 1–10 minute configurable lookback allows a slightly delayed IMAP reading to fill a RTSW gap
  • IMF clock angle: Computed as atan2(By, Bz) in degrees (0–360)
  • NZ timezone: Every minute includes time_nz with DST-correct Pacific/Auckland offset via Intl.DateTimeFormat
  • Source attribution: Each field carries a src label so the UI can show which satellite provided each data point

🚀 NASA DONKI Proxy

https://nasa-donki-api.thenamesrock.workers.dev/

Caches NASA DONKI data in KV and serves it to the browser. Required because NASA DONKI doesn't support CORS for browser requests directly. Also provides a human-readable HTML dashboard at the root URL.

How it works

  • Cron trigger fetches all 9 DONKI categories for the last 7 days, writes to KV with 24h TTL safety net
  • Blank write protection: If all categories return empty (NASA API blip), the cron skips the write — last good data remains
  • Date normalisation: All DONKI date strings (vary in format) normalised to ISO 8601 before storage

🔔 Push Notifications Worker

Manages push subscriptions, monitors forecast data, and sends Web Push notifications using VAPID authentication and AES-128-GCM encryption — all implemented natively without any third-party push library.

Subscription flow

  • User grants notification permission → browser creates a push subscription
  • App POSTs subscription + preferences + timezone to /save-subscription
  • Stored in KV keyed by SHA-256 hash of endpoint
  • Cloudflare request metadata (cf.latitude, cf.longitude) stored for location-aware filtering

Scheduled checks (cron)

  • Aurora forecast: Score from Forecast API via service binding. Fires when score crosses 40%, 50%, 60%, 80% thresholds.
  • Substorm activity: Status escalation QUIET → WATCH → LIKELY_60 → IMMINENT_30 → ONSET triggers increasingly urgent alerts.
  • X-ray flux: State machine (inactive → rising → peaked) tracking NOAA GOES X-ray data. Fires at M1, M5, X1, X5, X10 class thresholds and on confirmed peak.
  • IPS detection: Polls DONKI /GST. Fires when a new interplanetary shock event time is more recent than last known shock.

Broadcast system

Sends in batches of 40 subscriptions to stay within CPU time limits. Chains itself via self-fetch for large subscriber counts (up to 50 hops = 2,000 subscribers per event). Expired subscriptions (HTTP 410/404) automatically deleted.

Location-aware filtering

  • Aurora alerts apply the same latitude adjustment as the app — users further north only get notified if their adjusted score still crosses the threshold
  • Implausible latitudes (|lat| <30°) skipped for aurora/substorm alerts
  • Per-category opt-out stored per subscription
Crypto: Full VAPID JWT + AES-128-GCM web push encryption from scratch using the Web Crypto API. No external dependencies.

🌟 Aurora Sightings API

https://aurora-sightings.thenamesrock.workers.dev/

Stores and retrieves user aurora sighting reports. Entire sightings array stored as a single KV key — each GET or POST is just 1 KV read + 1 write.

GET — read all reports

  • Checks Cloudflare cache first — KV only read on cache miss
  • Response cached max-age=60

POST — submit a report

  • Validates: lat (number), lng (number), name (string, non-empty), status (8 valid values)
  • Name sanitised: trimmed to 50 chars, HTML tags stripped
  • Reads array → appends → prunes entries older than 24h → writes back
  • Invalidates Cloudflare cache for GET endpoint immediately after write

Cron cleanup

Daily cron writes an empty array to reset all sightings. Backup safety net — the POST pruning already handles continuous cleanup.

Rate limiting: Enforced client-side via localStorage (5-minute cooldown). Simple by design so offline-then-online scenarios still work.

🕳️ Coronal Hole History Worker

Stores coronal hole detection snapshots from the CME Modeller's client-side SUVI 195Å analysis. Maintains a SUVI frame index for the solar imagery animation viewer.

Endpoints

PathMethodWhat it does
/ch-historyGETAll snapshots in last 72 hours, sorted by timestamp
/ch-historyPOSTStore a coronal hole snapshot. Deduplicates within 10-minute windows.
/ch-history/framesGETSUVI 195Å frame URLs picked every 2 hours. Cached 30 min in KV.
/ch-history/statusGETDebug: index size, coverage hours, config values

Storage design

  • Each snapshot: ch-snapshot:{timestamp} with 96h TTL
  • ch-snapshot-index — sorted array of snapshot keys for fast retrieval
  • Max 144 snapshots (72h at 30-min intervals). Oldest evicted when limit exceeded.
  • Deduplication: new snapshot rejected if existing snapshot is within 10 minutes
  • Cron trigger refreshes SUVI frame index hourly by scraping NOAA 195Å directory
