System Overview

Five architectural layers, colour-coded throughout this document. Every arrow represents a real, confirmed data flow in the production codebase.

UserBrowser / PWA
📱 Progressive Web App

Installs to phone home screen with no app store. A service worker caches the UI shell for offline startup and handles push delivery when the app is closed.

📍 Optional GPS

If granted, latitude adjusts the displayed score ±0.2 % per 10 km from Greymouth. Also silently refreshed to the Push Worker 3 s after page load to keep location-aware alerts accurate. Never logged or shared.

🔔 Push Subscriptions

Browser generates a push endpoint + P-256 key pair via the Web Push API. Preferences (8 categories) stored locally and sent to the Push Worker KV on subscribe. Fully opt-in — nothing is sent before the user enables notifications.

📌 Aurora Sightings

Reports (naked eye / phone / DSLR / nothing / cloudy) submitted with optional GPS. One report per 5 min enforced via localStorage. Feed into the forecast "Now" slot and the live map.

ClientReact 18 browser app
🔄 Parallel Data Fetching

On load: Forecast API, Solar Wind IMF, GOES-18 Hp, GOES-19 Hp, NZ mag (EY2M), and IPS alerts fetched in parallel via Promise.allSettled. Substorm Risk Worker polled separately. All forecast data auto-refreshes every 60 s silently.

🧮 Client-Side Calculations

Newell coupling integral, P30/P60 substorm probabilities, location score adjustment, visibility text selection, substorm status classification — all computed in the browser from raw API responses. No server round-trip needed for display logic.

☀️ SUVI CH Detection

On the CME Visualization page: fetches GOES-19 SUVI 195 Å image via CORS proxy, draws to an off-screen 400 × 400 canvas, runs a 10-step coronal hole pipeline. Entirely client-side — no server compute involved.

🌌 3D Scene (60 fps)

Three.js r128. DBM propagation engine precomputes CME trajectories at 60 s resolution, then queries them in O(1) via binary-search interpolation. Parker spiral arms are parametric curves. Sun rotation uses ω = 2.61799 × 10⁻⁶ rad/s (27.27-day synodic period).

WorkersCloudflare edge
⚡ Forecast API

Cron every 5 min. Fetches solar wind, IMF, Hp, EY2M, IPS. Computes composite aurora score. Returns scored JSON + 24h history. KV-cached 30 s.

🔬 Substorm Risk

Per-request, KV-cached 60 s. Full Newell coupling, Bay onset detection, P30/P60 probabilities, solar loading score, all metrics.

🛰 IMAP Solar Wind

Per-request, 60 s cache. Merges IMAP-Hi L1 primary with DSCOVR RTSW fallback. Labels each data point with its source.

🔭 DONKI Proxy

Cron hourly. Caches CME, FLR, GST data from NASA. All clients served from KV to protect NASA rate limits.

🔔 Push Notifications

Cron every 5 min. Checks 8 alert categories. VAPID JWT + AES-128-GCM encryption from scratch via Web Crypto API. No npm dependencies.

🌑 CH History

Per-request + cron hourly. Stores CH snapshots (max 144 / 72 h). Maintains SUVI 195 Å frame index for backfill.

📌 Aurora Sightings

Per-request. Single KV key stores all reports as a JSON array. 24-h auto-prune. Edge-cached 60 s.

📢 Banner API

On demand. Admin-controlled sitewide banner. Also supports admin push broadcast to all subscribers.

Aurora Scoring Algorithm

The aurora score (0–100 %) is a composite metric computed by the Forecast Worker every 5 minutes and served from KV cache. It combines solar wind coupling physics, direct geomagnetic measurements, and hemispheric power into a single visibility probability estimate for the South Island of New Zealand.

Forecast WorkerCron: every 5 min
Primary Inputs
🌀 IMF Bz — the critical driver

The southward component of the Interplanetary Magnetic Field. When Bz is negative (pointing toward Earth's south magnetic pole) it drives magnetic reconnection at the dayside magnetopause, allowing solar wind kinetic energy to couple directly into the magnetosphere.

Both instantaneous Bz and rolling 10 and 30-minute averages are used. Sustained Bz below −5 nT for 15+ minutes is the strongest single predictor of aurora at subauroral latitudes. Brief southward excursions that immediately recover contribute much less.

⚡ Newell Coupling Function dΦ/dt

A physics-derived magnetospheric energy input rate, computed from L1 solar wind data in useForecastData.ts:

dΦ/dt = V^(4/3) · BT^(2/3) · |sin(θ/2)|^(8/3) V = solar wind speed (km/s) BT = sqrt(By² + Bz²) [total transverse IMF, nT] θ = atan2(By, Bz) [IMF clock angle]

Result divided by 1000 for normalisation. Higher values indicate more energy being injected into the magnetosphere per unit time. Averaged over 30 and 60-minute windows for substorm probability estimates.

🌎 Hemispheric Power (GOES Hp)

NOAA's GOES-18 and GOES-19 satellites directly measure the total auroral power being deposited into Earth's ionosphere in gigawatts, via particle precipitation measurements over the auroral zones. GOES-18 (primary) and GOES-19 (secondary) values are averaged. This is a direct measurement of how much aurora energy is being produced right now — not a model prediction from solar wind.

>20 GW: active. >50 GW: significant. >100 GW: major storm.

📡 NZ Geomagnetic dH (EY2M)

Eyrewell Observatory (Canterbury, NZ) 1-minute dH — rate of change of the horizontal magnetic field component. This is the only data source in the scoring system that is geographically specific to New Zealand. A sharp negative bay in dH (−50 nT or more over 15 min) indicates a substorm electrojet is active directly overhead, which means aurora is happening at that latitude right now.

This is why the NZ score can differ significantly from global Kp: Kp is a planetary average, EY2M tells us what's happening over Canterbury specifically.

Location Adjustment — Exact Formula
📍 GPS Latitude Correction

The Forecast Worker produces a baseScore referenced to Greymouth, NZ (latitude −42.45°). When GPS is granted, a finalScore is computed client-side and used for all display:

GREYMOUTH_LATITUDE = -42.45° dLat_rad = (user_lat - GREYMOUTH_LATITUDE) × π / 180 distance_km = |dLat_rad| × 6371 segments = floor(distance_km / 10) adjustment = segments × 0.2 % user_lat > GREYMOUTH_LATITUDE → finalScore = baseScore − adjustment user_lat < GREYMOUTH_LATITUDE → finalScore = baseScore + adjustment

Conservative by design: ±0.2 % per 10 km of latitude separation. Northland (further from the auroral oval) sees a lower score; Invercargill (closer) sees a higher one. The baseScore is stored in the score history array and sent to the Push Worker for server-side location filtering. finalScore is the adjusted value shown in the UI.

Substorm Risk Model

Substorm forecasting is a separate system from the headline score. It runs in the Substorm Risk Worker on every request (KV-cached 60 s) and provides near-real-time probability estimates with sub-5-minute latency.

Substorm Risk WorkerPer-request, 60 s cache
🌊 Newell Coupling Integral

The same Newell function computed in the client (see §03) is integrated over a rolling 30-minute and 60-minute window here using the L1 time series. This accumulated integral quantifies total magnetospheric loading over recent minutes — a stronger predictor of substorm onset than any instantaneous measurement.

dΦ/dt = V^(4/3) · BT^(2/3) · |sin(θ/2)|^(8/3) newell_now = current-point value newell_avg_30m = mean over last 30 min newell_avg_60m = mean over last 60 min

Historical threshold for significant NZ aurora activity: ~5–10 sustained over 30 min at typical solar wind speeds. This is used as input to the P30/P60 probability model below.

🔍 Magnetic Bay Onset Detection

A magnetic bay is the ground-level signature of a substorm: a sharp negative deflection of the horizontal geomagnetic field as the auroral electrojet activates overhead. The worker scans the EY2M dH time series for:

bay_onset = true if: max_drop > 50 nT AND drop occurs within any 15-min window in the recent EY2M data

If the bay flag is raised, substorm status advances directly to ONSET regardless of probability values. The auroral oval boundary is also shifted equatorward — this is fed into the visibility notification geometry.

📊 Probability Model — Exact Formulas

Probability of substorm onset in the next 30 and 60 minutes, computed client-side from Substorm Risk Worker metrics in useForecastData.ts:

base = tanh(0.015 × dΦ_15min_avg + 0.01 × dΦ_now) bzBoost = Bz < −3 nT : +0.10 Bz < −1 nT : +0.05 else : 0 P60 = clamp(0.25 + 0.6×base + bzBoost, 0.01, 0.90) P30 = clamp(0.15 + 0.7×base + bzBoost, 0.01, 0.90)

The tanh saturation prevents overconfident probabilities at extreme inputs. P30 is more sensitive to current coupling rate (higher 0.7 coefficient), while P60 gives more weight to sustained conditions (0.6 × averaged base). Probabilities are always clamped to [0.01, 0.90] — we never claim certainty.

CME Propagation Physics

The CME Visualization uses two propagation models. The full physics engine (heliosphericPropagation.ts) handles speed queries and colour transitions. A simpler empirical model handles scene positioning. Both run entirely in the browser at 60 fps.

DBM EngineheliosphericPropagation.ts
Drag-Based Model — Vršnak et al. 2013
Governing Equation

CMEs propagate through the heliosphere subject to aerodynamic-like drag from the ambient solar wind:

dv/dt = −γ · (v − w) · |v − w| v = CME radial speed (km/s) w = ambient solar wind speed (km/s) γ = drag parameter (km⁻¹)

Analytical solution for a decelerating CME (v₀ > w):

v(t) = w + (v₀ − w) / (1 + γ·(v₀−w)·t) r(t) = r₀ + w·t + ln(1 + γ·(v₀−w)·t) / γ

For an accelerating CME (v₀ < w):

v(t) = w − (w − v₀) / (1 + γ·(w−v₀)·t) r(t) = r₀ + w·t − ln(1 + γ·(w−v₀)·t) / γ

Inner boundary: 21.5 solar radii (≈0.1 AU) — DBM is only valid beyond the solar wind acceleration zone. All eruptions start at this point in the engine. Max propagation window: 10 days. Trajectory precomputed at 60 s timesteps; per-frame queries use O(log n) binary-search interpolation.

Refs: Vršnak et al. 2013, Solar Phys. 285:295 · Vršnak & Žic 2007, A&A 472:937

Drag Parameter γ — Per-CME Estimation

γ is estimated from the GCS cone parameters in the DONKI catalog. Exact formulas from estimateGamma():

speedClamp = clamp(speed, 200, 3000) km/s speedFactor = 1.8 × (400 / speedClamp)^0.55 → higher speed = lower drag (more massive) halfAngle = clamp(halfAngle, 10, 90) ° widthFactor = 0.7 + 0.6 × (halfAngle / 45) → wider CME = more drag (larger cross-section) γ = speedFactor × widthFactor × 10⁻⁷ km⁻¹ clamped to [0.05, 5.0] × 10⁻⁷ km⁻¹

Operational range 0.1–2.0 × 10⁻⁷ km⁻¹ is consistent with the statistical distribution from Vršnak et al. 2013 and Dumbović et al. 2021. The speed exponent of −0.55 reflects that faster CMEs are empirically less drag-affected per unit distance.

Refs: Cargill 2004, Solar Phys. 221:135 · Dumbović et al. 2021

Ambient Wind Speed Model

The background solar wind speed w is not a single constant — it is computed per-CME from three inputs in getAmbientWindSpeed():

1. Measured baseline: latest L1 speed from IMAP/DSCOVR (default 380 km/s if unavailable) 2. HSS contribution: for each detected coronal hole, compute angular separation to CME propagation path. If within influenceRadius = max(15°, CH_width × 0.7): temporalProfile(dtHours): 0–18h: ramp up (profile = dtHours / 18) 18–54h: plateau (profile = 1.0) 54–102h: decay (profile = linear to 0) angularWeight = exp(−0.5 × (dLon / (r×0.5))²) hssBoost = (peakSpeed − 380) × profile × weight w = max(w, 380 + hssBoost) 3. Preconditioning correction applied separately (see below)

The Gaussian angular weighting means a CME propagating directly through a CH's influence corridor sees the maximum HSS boost; those propagating at the edge see proportionally less. This physically represents how the fast solar wind stream has a finite angular width.

CME–CME Preconditioning & Compression

When multiple CMEs propagate along similar trajectories, two interaction modes are modelled:

Preconditioning — A leading CME sweeps through the heliosphere, creating a rarefied wake with reduced ambient density. A following CME propagating through this depleted medium experiences reduced drag:

dtEruption_sec = t_follower_start − t_leader_start preconditionDecay = exp(−dtEruption_sec / (3.5×86400)) (characteristic time τ ≈ 3.5 days) precondFactor = 1.0 − 0.5 × preconditionDecay → range [0.5, 1.0]; 0.5 = maximum 50% drag reduction γ_effective = γ × precondFactor

A follower erupting 1 day after a leader sees ~40% drag reduction. 3.5 days later: ~18% reduction. 7 days later: ~3% reduction. Temmer et al. 2017 found preconditioning lasts 2–5 days — this exponential model with τ=3.5 days sits in the middle of that range.

Compression — When a following CME is within 0.05 AU of a leading one (but has not yet overtaken it), the intervening plasma is compressed:

gap_km = leader_dist − follower_dist if 0 < gap_km < 0.05 AU (7.5M km): compressionIntensity = 1 − gap_km / (0.05×AU) densityBoost = max(existing, 1.0 + 2.0 × intensity)

This compression region causes the elevated density seen in the impact forecast when two CMEs arrive at Earth in close succession. Refs: Temmer et al. 2017, ApJ 835:141 · Werner et al. 2019, Space Weather 17:1

Scene Positioning — Simple Deceleration Model

While the DBM engine handles speed queries and colour transitions, the actual 3D position of each CME in the scene uses a simpler empirical formula from calculateDistanceWithDeceleration(). This avoids requiring the full engine for every CME mesh update:

a (m/s²) = 1.41 − 0.0035 × speed_km/s a (km/s²) = a_ms2 / 1000.0 If a ≥ 0 (accelerating / no decel): dist = u·t + 0.5·a·t² If a < 0 (decelerating): t_floor = (MIN_SPEED − u) / a [time to reach 300 km/s] If t < t_floor: dist = u·t + 0.5·a·t² Else: dist_decel = u·t_floor + 0.5·a·t_floor² dist_coast = 300 × (t − t_floor) dist = dist_decel + dist_coast Scene units = dist / AU_IN_KM × SCENE_SCALE AU_IN_KM = 149,597,870.7 SCENE_SCALE = 3.0

For Earth-directed CMEs that have a confirmed DONKI arrival time, the scene position is interpolated directly between the start time and arrival time — the model is not used. This ensures confirmed Earth-directed events hit Earth exactly when NASA predicts them to.

GCS Flux-Rope Geometry

Each CME is rendered as a particle cloud shaped like a GCS (Graduated Cylindrical Shell) flux rope. The geometry expands self-similarly as the CME propagates:

GCS_ARC_RADIUS_FRAC = 0.55 (arc/scale ratio) GCS_ARC_SPAN = π × 0.85 (≈153° half-arc) GCS_AXIAL_DEPTH_FRAC = 0.38 (depth vs lateral extent) lateral = max(dist × tan(halfAngle°), sunRadius × 0.3) sXZ = lateral / GCS_ARC_RADIUS_FRAC CME scale: (sXZ, sXZ × 0.38, sXZ)

The propagation direction is set by the DONKI longitude and latitude values. Each CME is an oriented Three.js mesh — the GCS arc opens perpendicular to the propagation vector. Particle positions are laid out along helical paths on the arc, with animated arrow textures showing the flux-rope structure.

Coronal Hole Detection Pipeline

Coronal holes are regions where the Sun's magnetic field is open to interplanetary space, allowing fast solar wind to escape. The app detects them automatically from the live GOES-19 SUVI 195 Å image, entirely in the browser. Every step runs as plain JavaScript — no server compute, no external detection service.

SUVI DetectorsuviCoronalHoleDetector.ts
10-Step Pipeline
1 · CORS Proxy Fetch

SUVI 195 Å image fetched via Cloudflare CORS proxy (/api/proxy/image) to produce a same-origin blob URL. Required because canvas.getImageData() throws a SecurityError on cross-origin images and NOAA does not include CORS headers on their image responses. Without the proxy, the entire pipeline is browser-blocked.

2 · Off-Screen Canvas

Drawn to an off-screen ANALYSIS_SIZE × ANALYSIS_SIZE = 400 × 400 px canvas. This resolution gives accurate polygon boundaries while keeping analysis time under 200 ms on typical mobile devices. A 512 px config option exists for sharper boundaries but is not needed for typical CH sizes.

3 · Gradient-Based Limb Detection

Scans outward from the disk centre in N_LIMB_ANGLES = 360 radial directions (one per degree). Finds the steepest negative brightness gradient in each direction — this locates the sharp photosphere limb boundary rather than the outer faint coronal halo, which extends visually beyond the physical disk. Produces a per-angle radius array with full 360° coverage.

4 · Disk Mask

A boolean per-pixel mask is built using the measured per-angle limb radius. Shrunk by a limb exclusion fraction to remove the immediate transition zone at the disk edge, which has brightness artefacts from the gradient transition and should not be included in the CH analysis region.

5 · Adaptive Median Luma

The median brightness (luma = 0.299R + 0.587G + 0.114B) of all pixels inside the disk mask. This adaptive baseline normalises for exposure variation, SUVI colour table changes, and day-to-day calibration drift. Using median rather than mean prevents active-region bright pixels from pulling the threshold too high.

6 · CH Candidate Flagging

Every disk pixel below the threshold is a CH candidate:

CH_DARK_THRESHOLD_FRAC = 0.52 candidate if: luma < 0.52 × disk_median_luma

Calibrated against a GOES-19 SUVI 195 Å reference image from 2026-03-12 showing a confirmed large elongated CH. The 0.52 value (raised from an earlier 0.40 default) captures the full extent of real CHs without including bright active regions or quiet-Sun background outside the disk.

7 · BFS Flood-Fill

Breadth-first search flood-fill connects adjacent candidate pixels into discrete regions. BFS (queue-based) is used rather than recursive DFS to avoid JavaScript call-stack overflow on large CH regions with thousands of pixels. Each connected component is a distinct CH candidate.

8 · Size Filter

Small regions discarded:

MIN_CH_PIXEL_FRAC = 0.003 keep if: region_pixels > 0.003 × disk_pixels

At 400 × 400 px with a disk covering ~75% of the canvas, this passes regions larger than ~360 pixels — approximately 1,000–2,000 MSH equivalent in area. Sub-threshold regions are instrumental noise or small ephemeral bright-point holes that would not drive meaningful HSS streams.

9 · Heliographic Projection

Region centroid and boundary polygon converted from pixel to heliographic (Carrington) coordinates using the standard solar orthographic projection:

x_norm = (px - cx) / r_limb [−1 to +1] y_norm = (py - cy) / r_limb [−1 to +1] lat = asin(y_norm) [heliographic lat] lon = atan2(x_norm, [Carrington lon] cos(lat))

No solar rotation correction is applied because SUVI 195 Å is in disk-centre coordinates — the Carrington system already rotates with the Sun.

10 · Output + Post

Returns CoronalHole[] with id, lat, lon, widthDeg, heightDeg, darkness (0–1 luma deficit fraction), estimated HSS speed, and polygon boundary. Empty array if no holes detected — no simulated fallback. Result is immediately posted to the CH History Worker for the 72 h archive and used to build Parker spiral arms in the 3D scene.

⚠️
Known detection limitations: CHs near the solar limb are foreshortened — their true area is underestimated. Polar holes are partially obscured by viewing geometry. Filament channels (dark elongated structures that are not open-field regions) can occasionally trigger false positives. CH boundaries detected from a single brightness threshold should be treated as approximate outlines, not precise maps.
HSS Speed Estimation & Parker Spiral

Each detected CH is assigned an estimated solar wind source speed at the surface, used both for the DBM ambient wind model and the Parker spiral geometry:

widthDeg = clamp(CH_angular_width, 5, 60) ° darkness = clamp(luma_deficit_fraction, 0, 1) peakSpeed = clamp( 500 + (widthDeg/60)×200 + darkness×120, 500, 900 ) km/s Parker spiral arm reach: 1.65 AU (extends beyond Earth orbit) SIR density peak in impact forecast: 18 h before speed rise

Larger, darker CHs drive faster streams — consistent with empirical studies correlating CH area and peak HSS speed at 1 AU. The 18-hour SIR density lead reflects the known structure of a Stream Interaction Region: compressed slow-wind plasma piles up ahead of the arriving fast stream.

Location-Aware Visibility Geometry — IGRF-13

For the three visibility tier notifications, the auroral oval equatorward boundary is estimated from Substorm Risk Worker Newell coupling averages, then compared to each subscriber's geomagnetic latitude:

// Oval boundary in geomagnetic latitude degrees newell = max(newell_avg_60m, newell_avg_30m × 0.85) boundary_gmag = −(65.5 − newell / 1800) clamped to [−76°, −44°] geomagnetic lat bay_onset flag → further clamp to −47.2° // Geographic → geomagnetic lat conversion (IGRF-13) POLE_LAT_RAD = 80.65° × π/180 POLE_LON_RAD = −72.68° × π/180 (geographic coords) sin(gmag_lat) = sin(geo_lat) × sin(POLE_LAT_RAD) + cos(geo_lat) × cos(POLE_LAT_RAD) × cos(geo_lon − POLE_LON_RAD) // Visibility horizon extends equatorward of boundary visDeg = 9.0 + (score/100) × 16.0 → range [9°, 25°] depending on activity level visHorizon_gmag = boundary_gmag + visDeg distToVis = subscriber_gmag − visHorizon_gmag // DSLR tier fires when: distToVis ≤ −3° (horizon 3° past observer) // Phone tier fires when: distToVis ≤ −1° // Naked tier fires when: distToBoundary ≤ 5° // All thresholds moon-adjusted (see categories table)

The IGRF-13 dipole pole position (80.65°N, 72.68°W geographic) is used for the geomagnetic latitude conversion. This is the standard approach used in operational aurora forecasting — it correctly accounts for the fact that the geomagnetic and geographic poles do not coincide, which matters significantly for New Zealand's longitude.

Push Cryptography — RFC 8291 Web Push from Scratch
🔐 VAPID JWT (RFC 8292)

Every push request is authenticated with a signed JWT identifying the push server to browser push services (FCM, APNs web, Mozilla):

Header: { typ:"JWT", alg:"ES256" } Payload: { aud: push_endpoint_origin, exp: now + 12h, sub: VAPID_SUBJECT } Signature: ECDSA P-256 using VAPID_PRIVATE_KEY (stored in Worker Secrets — never in KV or logs) Authorization: vapid t={jwt}, k={VAPID_PUBLIC_KEY}

The JWT is generated per delivery using crypto.subtle.sign({ name:'ECDSA', hash:'SHA-256' }). The public key is baked into the client app at build time. No third-party VAPID library — implemented using the Web Crypto API directly.

🔒 Payload Encryption (RFC 8291)

Push payload is AES-128-GCM encrypted using the subscriber's P-256 public key stored at subscribe time:

1. Generate ephemeral ECDH P-256 server keypair 2. ECDH: deriveBits(serverPrivKey, subscriberPubKey) → ecdhSecret (32 bytes) 3. PRK = HMAC-SHA256(auth_secret, ecdhSecret) 4. keyInfo = "WebPush: info" || 0x00 || uaPubRaw || serverPubRaw 5. IKM = HMAC-SHA256(PRK, keyInfo || 0x01) 6. salt = randomBytes(16) 7. PRK2 = HMAC-SHA256(salt, IKM) 8. CEK = PRK2("Content-Encoding: aes128gcm") [16B] 9. NONCE = PRK2("Content-Encoding: nonce") [12B] 10. Encrypt payload with AES-128-GCM(CEK, NONCE) 11. Send: salt || record_size || serverPubRaw || ciphertext

Entire key derivation and encryption uses crypto.subtle. No npm dependencies. Validated against RFC 8291 test vectors. This means push payloads are end-to-end encrypted between the Push Worker and the user's browser — the push service (FCM etc.) cannot read the notification content.

📦 Batch Delivery + Cleanup

Subscriptions processed in batches of 40 to stay within Cloudflare Worker CPU limits. For larger subscriber counts, the worker chains itself via self-fetch:

BATCH_SIZE = 40 subscriptions per invocation MAX_CHAIN = 50 hops maximum → supports up to 2,000 subscribers per alert HTTP 410 or 404 from push service → subscription auto-deleted from KV immediately

Stale subscriptions (from browser reinstalls, revoked permissions, or expired endpoints) are silently purged on the first delivery failure. This keeps the subscriber list clean without requiring a separate maintenance job.

⏱ Cooldown System

Multi-level cooldown system prevents notification spam:

Per-topic global cooldown: COOLDOWN_{topic} key in KV TTL = cooldown_minutes for that category Per-subscriber visibility cooldown: COOLDOWN_vis_{tier}_{subscriber_id} key in KV TTL = 2 hours per tier Escalation-only rule: Visibility tiers only fire on improvement (DSLR→phone→naked, never same tier twice) Conditions must drop fully before re-triggering

The two-hour per-subscriber cooldown means a subscriber who receives a "DSLR visible" notification won't receive another DSLR notification for 2 hours even if conditions drop and recover. This significantly reduces notification fatigue during fluctuating conditions.

Backend Workers — Detailed Reference

All backend logic runs on Cloudflare Workers — serverless JavaScript running at the edge globally. No persistent file system; all state is in Cloudflare KV. Worker URLs are not published to prevent endpoint abuse. Every worker has CORS headers allowing access from the app's domain only.

Forecast APICron: 5 min
What It Does

On cron: fetches IMAP solar wind, NOAA GOES-18/19 Hp, EY2M geomag, and IPS shock events. Runs the composite aurora score algorithm. Writes result to KV with a 30 s client-facing TTL. Clients never hit external APIs directly — they read from KV only. Score history (24 h array) accumulates in KV and is trimmed on each write.

Response Shape

baseScore (0–100, Greymouth reference), all raw inputs (Bz, Bt, speed, density, Hp, EY2M dH), 24 h history array with timestamps, moon phase/times, solar event flags, IPS shock flag, lastUpdated epoch. Clients compute finalScore (location-adjusted) from baseScore locally.

Substorm RiskPer-request, 60 s cache

Fetches fresh L1 solar wind and EY2M data on every request (result KV-cached 60 s). Computes the full Newell integral over 30 and 60-min windows, geomagnetic bay detection, solar loading score (a composite of active-region flare probabilities and hemispheric power trends), substorm status, and P30/P60 probabilities. Returns the complete metrics object including every intermediate value.

Why separate from the Forecast API? The substorm bay onset can develop in under 5 minutes. The Forecast Worker runs on a 5-min cron — there could be up to 5 minutes of lag before it reflects a rapid onset. The Substorm Risk Worker is per-request (with 60 s cache), so the status it returns is always within 60 seconds of current conditions. This difference is meaningful for real-time bay detection.

IMAP Solar WindPer-request, 60 s cache

Merge strategy: Attempts IMAP-Hi L1 data first. If unavailable or stale (IMAP is a newer NASA mission and data gaps occur), falls back to NOAA DSCOVR RTSW. Each data point in the 24 h returned time series is individually labelled with its source — IMAP or NOAA RTSW — shown on the IMF chart.

L1 travel time reminder: All L1 measurements shown in the app reflect conditions that are currently 45–60 minutes from arriving at Earth at typical solar wind speeds. A southward Bz visible in the IMF chart means aurora may develop within the hour — not that it is happening right now. The "15 min" forecast slot accounts for the current L1 reading; the "Now" slot is based on the geomagnetic/Hp measurements already at Earth.

DONKI ProxyCron: hourly

Fetches CME catalog, FLR list, and GST shock events from NASA DONKI on an hourly cron. Stores in KV. All client requests are served from KV — no browser ever calls NASA directly. This protects the shared NASA API key from being rate-limited by multiple concurrent users. The NASA API key is stored in Worker Secrets and is never present in any client-facing response.

CME processing: For each CME, the "most accurate" GCS analysis is selected (or the first if no most-accurate flag). Only CMEs with valid speed, longitude, and latitude values are included in the modellable set. Earth-directed classification: |longitude| < 45°. Predicted arrival time extracted from linked GST activityID if present.

CH HistoryCron: hourly
Snapshot Storage

Accepts CH detection snapshots POSTed by the browser. Deduplicates within 10-minute windows (rejects a new snapshot if one already exists within ±10 min). Stores with 96 h TTL. Sorted index key for fast range queries. Evicts oldest entry when max 144 snapshots is exceeded.

SUVI Frame Index

Hourly cron scrapes the NOAA SUVI 195 Å animation directory HTML listing. Parses all PNG filenames and their ISO timestamps. Selects one frame per 2-hour window (the first frame in each window). Caches result for 30 min. Used by the browser to backfill CH history gaps by reprocessing historical SUVI images through the same detection pipeline.

Banner APIOn demand

Admin-controlled sitewide banner (BANNER_AUTH_TOKEN in Worker Secrets). Stores content in KV. Fetched on every page load — shown as a strip above the app content. The dynamic substorm alert visible in the app's global header is separate: it is generated entirely client-side from the Substorm Risk Worker response.

Aurora Sightings

User-submitted reports provide real-world ground truth that solar wind models cannot. "Nothing" reports are explicitly valuable — they calibrate when the model over-predicts.

What You Submit
8 Status Values

Negative sightings ("Nothing") explicitly confirm clear-sky absence of aurora when the model predicts activity — this is calibration data, not noise. Cloudy reports confirm obstructed sky rather than absent aurora.

Rate Limiting + Privacy

One report per 5 minutes via localStorage timestamp. This works offline-then-online correctly, unlike server-side IP limiting which fails for mobile users on carrier NAT. Name trimmed to 50 chars, HTML stripped. GPS coordinates are stored only in the sighting record and used only to position the map dot — not linked to any persistent identity.

Sightings Worker
Storage Design

All reports in one JSON array under a single KV key. Each POST: read → append → prune entries older than 24 h → write back. This is exactly one KV read + one write per submission — very cheap. GET responses served from Cloudflare edge cache with Cache-Control: max-age=60. Daily cron resets the array as a safety net. Write invalidates the edge cache immediately so new reports appear globally within 1–2 seconds.

Use in Forecast

Reports from the past 30 minutes within approximately 200 km of the user's location are counted. If ≥2 naked-eye reports exist nearby, the "Now" visibility slot shows a sightings-confirmed message alongside the model prediction. Model prediction and sightings are displayed as separate, independent information — sightings never override or suppress the score.

Transparency & Limitations

Aurora forecasting is hard. This section is a plain-English account of the known limits of each part of this system.

Known Limits
⏱ L1 Travel Time Uncertainty

Solar wind data is measured at L1, approximately 1.5 million km from Earth. At 400–600 km/s, the data shown in the IMF charts is 45–60 minutes away from arriving at Earth. We cannot know if those conditions will persist, intensify, or die out before arrival. The "15 min" slot accounts for this lag; the uncertainty cannot be eliminated.

🌎 Ionospheric Variability

Identical solar wind conditions produce different aurora on different nights. Ionospheric conductivity, substorm phase, ring current state, and magnetospheric pre-conditioning all affect the visible outcome. The score is a probability estimate — not a deterministic forecast. A score of 60% means significant aurora is more likely than not, not that it is guaranteed.

🌑 CH Detection Accuracy

Single brightness-threshold detection calibrated to one reference image. CHs near the solar limb are foreshortened. Polar holes are partially obscured. Filament channels may false-positive. Detected boundaries should be treated as approximate outlines. Multiple detections over 72 h provide better confidence than any single frame.

🚀 CME Trajectory Deflection

CME trajectories can deflect significantly between Sun and Earth due to interaction with the background solar wind, helmet streamer belts, and other CMEs. A CME modelled as Earth-directed from DONKI GCS parameters may still miss Earth. Conversely, a flank CME can produce aurora through its compressed sheath. The visualizer shows the DONKI catalog parameters as-is — no deflection model is included.

📊 Score ≠ Kp

The aurora score is a custom index tuned for South Island New Zealand visibility. It is not Kp. Kp is a 3-hourly global planetary average across all latitudes; this score uses sub-minute data from local NZ instruments and is calibrated for the subauroral latitude band. The score responds faster to changing conditions than Kp and can read higher because local EY2M data is incorporated directly.

🔔 5-Minute Notification Lag

Notification checks run every 5 minutes on a cron. A rapid onset can go from quiet to naked-eye visible in under 5 minutes — faster than the check cycle. The "overnight watch" notification around sunset is the practical hedge: it provides advance notice so users are already watching when conditions peak.

📐 DBM Model Uncertainty

The drag parameter γ is estimated from CME speed and half-angle — the actual value depends on CME mass, cross-section, and ambient density, all of which are uncertain from remote sensing. The DBM arrival time uncertainty is typically ±6–12 hours at 1 AU. For confirmed Earth-directed events with a DONKI-supplied arrival time, the catalog value is used instead of the model estimate.

🌐 Data Source Outages

All data sources can become unavailable. IMAP is a relatively new mission with occasional data gaps — DSCOVR fallback is automatic. NOAA SWPC occasionally has maintenance outages. The app is built to degrade gracefully: missing data shows stale values with timestamps rather than crashing. The impact forecast uses a geometry-based fallback when the propagation engine hasn't built yet.
